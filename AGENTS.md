# tekton-pipeline-queue - AI Agent Guide

A kubebuilder (go.kubebuilder.io/v4) operator that queues Tekton PipelineRuns.
It admits runs created with `spec.status: PipelineRunPending` per *lane* (a key
derived from label values) under a `concurrency` limit and a `strategy`
(`Queue`, `ReplaceQueued`, `CancelInProgress`). There is no stored queue — the
live PipelineRun set is the single source of truth. See `README.md` and
`docs/` for the full concept.

```
config/samples/*                            Example CRs (edit these)

deploy-templates/README.md                  Generated by `make helm-docs` from README.md.gotmpl (DO NOT EDIT)
docs/api.md                                 Generated by `make api-docs` (crdoc) (DO NOT EDIT)
docs/{use-cases,debugging}.md               Hand-written docs (edit these)

ct-configs/                                 chart-testing (ct) lint config for the Helm chart
Makefile                                    Build/test/deploy commands

```

Single-group layout, no webhooks. Two controllers are registered in
`PROJECT`: the `PipelineRunQueue` API (group `edp`, domain `epam.com`,
`v1alpha1`) and an external-type watch on `tekton.dev/v1` PipelineRuns
(`github.com/tektoncd/pipeline`). The operator never installs or owns the
Tekton CRDs.

## Critical Rules

### Never Edit These (Auto-Generated)

- `config/crd/bases/*.yaml` — from `make manifests`
- `deploy-templates/crds/*.yaml` — synced from `config/crd/bases` by `make manifests`
- `config/rbac/role.yaml` — from `make manifests`
- `docs/api.md` — from `make api-docs` (runs as part of `make manifests`)
- `deploy-templates/README.md` — from `make helm-docs` (edit `README.md.gotmpl` instead)
- `**/zz_generated.*.go` — from `make generate`
- `PROJECT` — from `kubebuilder` CLI

CI runs `make validate-docs` — it fails if `docs/api.md` or
`deploy-templates/README.md` are stale.

### Never Remove Scaffold Markers

Do NOT delete `// +kubebuilder:scaffold:*` comments. The CLI injects code at
these markers.

### Keep Project Structure

Do not move files around; use `kubebuilder create api`/`kubebuilder edit` for
scaffolding changes rather than creating scaffold files manually.

### E2E Tests Run Against the Current Kubeconfig Context

`make e2e` is cluster-agnostic: it runs the full Chainsaw suite against
whatever kubeconfig context is current, and the scenarios install their own
prerequisites. Use `make start-kind` first for a dedicated
[Kind](https://kind.sigs.k8s.io/) cluster (same as GitHub Actions CI); the
core-cluster pipeline switches context to a disposable vcluster instead.
ALWAYS verify the current context before running — never point it at a real
dev/prod cluster.

## Testing & Development

```bash
make build             # Linux manager binary → dist/manager-$(GOARCH); the
                       # Dockerfile only copies it (no in-image compilation)
make test              # Unit tests (envtest; Ginkgo + Gomega, see suite_test.go)
make run               # Run locally against current kubeconfig context
make start-kind        # Create the e2e Kind cluster (hack/kind-$(KUBE_VERSION).yaml)
make e2e               # Chainsaw e2e: builds+loads the image, exports the e2e
                       # env contract, runs tests/e2e/chainsaw
make delete-kind       # Delete the e2e Kind cluster
```

The Chainsaw suite is self-provisioning and runs as one unit: the
`00-install` test runs first (lexical order) and installs Tekton Pipelines
(idempotently) plus the operator chart, using the
`CONTAINER_REGISTRY_URL`/`CONTAINER_REGISTRY_SPACE`/`E2E_IMAGE_REPOSITORY`/
`E2E_IMAGE_TAG` env vars — the same contract the `e2e-chainsaw` Tekton task
provides when running the suite in a vcluster on the core cluster. Never
run scenarios individually; they assume `00-install` has run.

E2E scenarios live in `tests/e2e/chainsaw/<behavior>/` (fifo-admission,
concurrency, lane-isolation, replace-queued, cancel-in-progress,
completion-promotion, crd-validation, invalid-selector). Add a new directory
per behavior — provisioning lives only in `00-install`; see
`tests/e2e/chainsaw/README.md`.

## Design Notes

### API Design

**Markers in `api/v1alpha1/pipelinerunqueue_types.go`:**

```go
// +kubebuilder:object:root=true
// +kubebuilder:subresource:status
// +kubebuilder:printcolumn:...
// +kubebuilder:validation:...  // on fields
// +kubebuilder:default=...     // on fields
```

- Use `metav1.Condition` for status conditions, `metav1.Time` for timestamps
- Follow K8s API conventions (`spec`, `status`, standard field names)
- Queue annotations stamped on PipelineRuns are defined in
  `api/v1alpha1/annotations.go` — keep README/docs in sync when changing them

### Controller Design

The reconciler is intentionally stateless: every reconcile re-derives lanes
from the live PipelineRun set (`bucketLanes` in `lane.go`), then applies the
strategy (`strategy.go`). Admission clears `spec.status`; cancellation sets
`CancelledRunFinally`. Both patches also stamp the queue annotations in the
same request.

**RBAC markers (in `pipelinerunqueue_controller.go`) — this operator is
read-only on its own CRD spec and only patches PipelineRuns:**

```go
// +kubebuilder:rbac:groups=edp.epam.com,resources=pipelinerunqueues,verbs=get;list;watch
// +kubebuilder:rbac:groups=edp.epam.com,resources=pipelinerunqueues/status,verbs=get;patch
// +kubebuilder:rbac:groups=tekton.dev,resources=pipelineruns,verbs=get;list;watch;patch
```

**Implementation rules:**

- **Idempotent reconciliation**: safe to run multiple times; no stored queue state
- **Predicate discipline**: PipelineRun watch events are filtered by
  `pipelineRunPredicate` — only bucket transitions and label changes trigger
  reconciles. Keep this in mind when adding fields the decision depends on.
- **Structured logging**: `log := log.FromContext(ctx); log.Info("msg", "key", val)`
- **Metrics**: update `metrics.go` (and its tests) when lane/queue behavior changes

### Logging

Follow Kubernetes logging message style guidelines:

- Start with a capital letter; no trailing period
- Active voice, past tense (`"Could not delete Pod"`, not `"Cannot delete Pod"`)
- Name the object type (`"Admitted PipelineRun"`, not `"Admitted"`)
- Balanced key-value pairs

**Reference:** <https://github.com/kubernetes/community/blob/master/contributors/devel/sig-instrumentation/logging.md#message-style-guidelines>
